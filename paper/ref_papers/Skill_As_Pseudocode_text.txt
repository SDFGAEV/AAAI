Refactoring Skill Libraries to Pseudocode for LLM Agents

Abstract
Markdown skill libraries for LLM agents ship
as free-form prose, forcing the agent to re-
derive both the input schema and the concrete
invocation syntax on every retrieval. We ob-
serve that this often produces a “confused →
re-retrieve →still confused” loop in which the
agent issues a partially-correct action, receives
uninformative environment feedback, and re-
retrieves the same prose. We propose Skill-as-
Pseudocode (SaP), an automatic conversion of
markdown skill libraries into typed pseudocode
with deterministic quality control. For each
cluster of similar procedural passages drawn
from one or more skills, SaP extracts a typed
contract and filters it through a four-check de-
terministic verifier (coverage, binding, replace-
ment, risk). Promoted contracts are inlined into
a rewritten skill skeleton together with restored
concrete action templates, giving the agent two
complementary signals: a typed signature for
what the skill does and a concrete template for
how to invoke it. On the 134-game ALFWorld
unseen split with gpt-4o-mini, pooled across
three seeds, SaP wins 82/402 paired games
versus 47/402 for the Graph-of-Skills (GoS)
baseline (pooled McNemar p = 8.2×10−5), at
−22.8 ± 6.4% input tokens and −14.5 ± 4.1%
LLM calls per game.

Introduction
Modern LLM-based agents (Yao et al., 2023;
Schick et al., 2023) rely on skill libraries—
collections of pre-authored procedural documents
the agent retrieves and follows at task time. Typed
catalogs such as OpenAPI specifications (Ope-
nAPI Initiative, 2024; Song et al., 2023; Qin et al.,
2024; Patil et al., 2024) already expose machine-
readable function signatures, but the more com-
mon deployment surface for agent skills today is
markdown: Anthropic’s SKILL.md documents (An-
thropic, 2025), the Graph-of-Skills (GoS) library
* Corresponding authors.
used as our experimental substrate, and server-side
MCP descriptions (Anthropic, 2024) all ship as
free-form prose for human and LLM readers.
Prose is the bottleneck.
We argue, and our ex-
periments support, that the prose representation
itself, not the absence of factoring, is the dominant
bottleneck when LLM agents consume markdown
skill libraries. A markdown skill document does
not separate what the skill produces (its output
schema) from how the skill is invoked (the con-
crete primitive actions the environment accepts),
and the two are often interleaved with prose hints,
caveats, and worked examples. At retrieval time
the agent must re-derive both on every call. Em-
pirically this produces a recurring retrieval-action
feedback loop on ALFWorld: the agent issues a
SkillRequest, reads back a long markdown body,
emits an action whose verb or argument is slightly
off (put X in/on Y rather than move X to Y;
heat mug before holding the mug), the environ-
ment returns “Nothing happens.”, and the agent
re-retrieves looking for the missing detail (§ 6.1).
The same loop drives both task failures and the
per-game LLM-call cost.
Typed pseudocode is a better representation.
Two pieces of information short-circuit the loop.
(1) A typed contract that exposes the skill’s trig-
ger, input schema, output schema, and pre/post-
conditions tells the agent in one glance what it
can do and what arguments it must supply; this
is the same affordance that OpenAPI specs give
typed-routing agents. (2) The concrete action tem-
plates (“go to {recep}”, “heat {obj} with
{appliance}”) that the contract abstracts over tell
the agent how to actually invoke the underlying
environment, in the exact tokens the environment
accepts. Together these two pieces form a struc-
tured pseudocode spec: a typed signature paired
with template-level concrete syntax. Markdown
skill libraries today contain both pieces of informa-

Skill-as-Pseudocode (SaP).
We propose to auto-
mate the prose-to-pseudocode conversion of mark-
down skill libraries with deterministic quality con-
trol.1 Each original markdown skill (which we
call a parent skill because the rewrite layer inlines
child contracts beneath it) parses into a sequence
of procedural units—markdown sections under a
heading, each describing one sub-procedure (Li
et al., 2023b). Units across parents with similar
verbs, objects, and code linkage are grouped into
a candidate cluster of passages hypothesised to
describe the same underlying procedure. For each
cluster, SaP (i) drafts a typed contract κ with a
single LLM call (Stage 3), (ii) runs a four-check
deterministic verifier on κ for Coverage, Binding,
Replacement, and Risk (§ 3.3), and (iii) rewrites
each parent’s matching units into an invoke(κ,
args) placeholder grounded by per-call argument
bindings. Two LLM-aware post-verifier passes—
Binding Extraction (BE) and Rewrite Cleanup
(RC)—ground the placeholder per call-site and re-
move residual prose conflicts; both are validated
by deterministic post-checks (contracts remain ref-
erenced; required inputs have bindings).
At retrieval time the agent receives a substituted
bundle combining the rewritten parent skeleton
with invoke(κ, args) placeholders, the inlined
typed contract for each invoked κ, and the restored
concrete action templates the placeholder stands
for—delivering the what (contract) and how (tem-
plate) signals in a single response.
Cross-parent factoring is a secondary benefit,
not the central claim.
Contracts callable in mul-
tiple parents are promoted to standalone child skills
(factoring is real, § 5.1), but the retrieval-pool abla-
tion (§ 5.5) shows the agent’s gain comes from the
parent-side substituted bundle, not the child library:
surfacing children as standalone retrieval results
drops reward by 27%.

3
Method
3.1
Problem statement
A skill library L = {p1, . . . , pn} is a set of par-
ent skills, each a procedural document (markdown
SKILL.md, OpenAPI specification, or MCP server
description). Each parent pi consists of a sequence
of procedural units ui,1, . . . , ui,ki (markdown sec-
tions, OpenAPI parameter groups, etc.). A candi-
date cluster c is a set of procedural units, drawn
from one or more parents, hypothesised to describe
the same underlying procedure in different prose.
A typed contract κ is a structured pseudocode
spec
with
explicit
trigger,
input_schema,
output_schema,
preconditions,
post-
conditions,
and side_effects fields (full
schema in Appendix F).
κ is the structured
surrogate for the prose in c: the same information
re-expressed in a form the agent can consume in
one read instead of re-deriving it on every retrieval.
We define the conversion c →κ as valid iff κ
satisfies four deterministic checks (§ 3.3): Cov-
erage, Binding, Replacement, and Risk. The
verifier returns either “valid, with witness κ and
supporting evidence” or “invalid, with check-level
rejection reasons”. The verifier produces no soft-
classifier score; any rejection is grounded in one
of four named checks. Given a set K of validated
contracts, the converted library L′ rewrites each
parent’s matching units into invoke(κj,⃗a) place-
holders and adds each κj as a child skill (reached
via invoke, not direct retrieval; § 3.5). Cross-
parent factoring (when |src(κ)| > 1) is a sec-
ondary benefit; the principal benefit is the prose-to-
pseudocode conversion itself.

3.3
The four verifier checks
Given a candidate c and contract draft κ, the veri-
fier computes four rule-based, check-level scores;
unlike learned step-level verifiers (Lightman et al.,
2024; Uesato et al., 2022), the output is a struc-
tured rejection profile, not a soft classifier score.
The four checks: Coverage (token recall of κ’s
trigger/I-O strings against each parent’s unit text,
catches misnamed contracts); Binding (for each
required input, fraction of (parent, input) pairs
whose unit text overlaps the input name, catches
over-wide clusters); Replacement (fraction of par-
ents whose unit admits an invoke(κ)-substitution
preserving markdown structure, catches control-
flow entanglement); Risk (AST scan of contract
scripts/resources for unsafe sinks like rm -rf or un-
declared network egress, hard-rejects at weighted
score ≥0.80). Each check catches a distinct class
of would-be-promoted candidates—no single layer
accounts for more than ∼51% of first-failed rejec-
tions and the layers are nearly disjoint (Appendix I,
Table 7). Full threshold definitions and the per-
layer rejection bucket are in Appendix I.
3.4
Decision policy + calibration (summary)
The four scores are combined into a scalar promo-
tion score s(ϕ) = wb binding + wc coverage +
wr replacement−ws risk and mapped to a three-
tier decision (auto_promote, review, reject) by
thresholds (τauto, τrev). We calibrate the two thresh-
olds against 30 synthetic negative controls from
three single-library classes (same-domain-distinct,
near-miss verb-object permutations, swapped-
contract pairings; n = 10 each), a benchmark-free
counterpart to output calibration (Guo et al., 2017)
and Chow-style reject options. On skills_500 a

Experimental Setup
4.1
Library and benchmarks
Skill library.
We use the public Graph-of-Skills
skills_500 skillset (Liu et al., 2026): 500 mark-
down skill documents, 5,709 procedural units after
parsing, 54 skills shipping with an executable script.
The skillset covers ∼10 domains (37 alfworld-*
embodied-agent skills, 8 *-automation API wrap-
pers, ffmpeg-*, senior-* software engineering,
mathematical and signal-processing skills, etc.).
We selected skills_500 as the smallest GoS
skillset whose content matches the ALFWorld
benchmark domain: skills_200 (Appendix A)
contains zero alfworld-* skills (verified by grep)
and gives the same near-zero reward across all re-
trieval modes on ALFWorld.
ALFWorld.
The unseen split of ALFWorld
(Shridhar et al., 2021), 134 text-game household
tasks (heat-and-place, cool-and-place, clean-and-
place, pick-and-place, look-at-object, etc.). Each
task is a partially observable environment; success
is a binary reward at the end of an episode of up to
30 environment-action steps (skill-retrieval turns
do not consume the action budget).
SkillsBench (subset).
A 10-task subset of the 87
Docker-environment coding tasks in SkillsBench
(Li et al., 2026): 6 easy + 4 medium (full task
names in Appendix E). The full 87-task evaluation
was deferred for budget; sample bias is discussed
in § 5.3 and Limitations.

6.1
Mechanism: structure breaks the
retrieval-action loop
SaP’s bundle delivers two structured signals (typed
contract + inlined action template) in one response,
while GOS forces the agent to locate both pieces
inside a long mixed prose body—an instance of the
“lost-in-the-middle” degradation (Liu et al., 2024a).
Three observations support this: post-retrieval cor-
rectness (Table 4), a per-game loop signature (Ap-
pendix N), and a trace-level case study.

6.2
What BE and RC each repair
The +5 pp/+5 pp component ablation (Table 3) iso-
lates two failure modes already detailed in § 5.4:
RC fixes parent-residual verb conflicts (e.g., an
Example block emitting put X in/on Y where the
engine accepts only move X to Y); BE drops 30%
of deterministic call-sites as spurious. Both passes
target distinct upstream causes of the same prose-
induced loop: stale syntax in unrewritten content
(RC) and false-positive cluster membership (BE).
7
Conclusion
Our results raise a broader question: should skill
libraries be authored in pseudocode directly? Mark-
down optimises for human authors, but LLM agents
pay the prose-to-structure conversion cost on ev-
ery retrieval. SaP shows this cost is recoverable
post hoc with deterministic quality control—paid
at index time rather than at every read. Publishers
in typed catalogues (OpenAPI, MCP) can there-
fore pre-substitute at index time; domain experts
who cannot write pseudocode directly keep writing
prose while the verifier-driven pipeline maintains
an agent-readable pseudocode layer on their be-
half. More broadly, SaP’s discipline—each LLM
extraction is a hypothesis checked by deterministic
rules, not a black-box output—transfers to other
prose-to-structure tasks (schemas from runbooks,
terms from legal docs, API specs from wikis). The
four checks vary per domain; the rule generalises:
accept with a witness, reject with a named cause.
